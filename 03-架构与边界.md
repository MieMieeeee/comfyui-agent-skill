# 架构与边界

## 当前架构方向

本项目当前阶段应采用 **skill-first 的 MVP 路线**。

这意味着：

- 当前不要做 MCP
- 当前不要做通用平台
- 不要过早做大而全抽象
- 要做的是一个小而可复用的执行核心，供未来 skill 层调用

## 推荐分层

虽然这是一个小型 MVP，但代码仍应尽量保留基本的职责分离。

### 1. 工作流执行层
负责：

- 加载 workflow
- 写入 prompt 输入
- 如有需要，写入固定尺寸参数
- 通过现有封装调用 ComfyUI
- 收集返回的图片字节
- 保存输出文件
- 返回统一结果对象

### 2. 面向 Skill 的入口层
负责：

- 对外暴露一个更高层的函数，例如 `text_to_image(prompt: str)`
- 保持调用方语义清晰
- 避免 workflow 细节在外部到处泄漏

### 3. 共享模型 / 结果结构
负责：

- 定义统一结果结构
- 如有必要，定义基础错误结构
- 为未来扩展稳定返回值形式

## 重要边界规则

### 应优先复用的部分
请尽量复用当前仓库已有能力，尤其是：

- `ComfyApiWrapper`
- `ComfyWorkflowWrapper`
- `run/get_zimage.py` 中已经验证通过的执行方式

### 当前不应做的事情
请**不要**：

- 重构整个仓库
- 用全新框架替换当前 wrapper
- 在第一步就引入复杂 registry 系统
- 增加不必要的抽象层
- 把当前任务做成一个通用工作流引擎

### 当前阶段的配置哲学
当前阶段配置可以非常轻量。

可以接受：

- 第一个 workflow 固定
- 输出节点固定
- 宽高固定

如果增加一个很小的配置文件能够带来明显收益，可以接受。  
但不要在第一阶段把配置系统本身做成一个大型子系统。

## 后续扩展方向

最终代码应便于后续扩展：

- 增加第二个 workflow
- 增加图生图能力
- 增加一个薄 skill adapter
- 增加轻量 workflow 配置
- 将来再抽象出更稳定的能力层

但这些未来可能性不应主导第一阶段实现。
